# AniGroom Stage 1 Technical Framework

This document defines the confirmed Stage 1 framework for the formal
AniGroom implementation. It is the technical basis for the next implementation
stage. It is not an experiment log and it is not a place for speculative
modules.

## 1. Scope

Stage 1 learns editable, mesh-rooted animal fur from multi-view image
supervision.

The first target asset is the white tiger. The output of Stage 1 is a static
reconstructed groom represented by roots on the mesh surface, explicit groom
parameters, and anisotropic Gaussian strand samples rendered by `gsplat`.

The goal of Stage 1 is reconstruction quality and training correctness. Asset
extraction objectives can be added later, but they must not pollute the Stage 1
reconstruction specification.

## 2. Non-Negotiable Engineering Rules

AniGroom must not be developed by closed-door reinvention.

For each major module, the implementation must identify the closest reliable
reference implementation or paper and state:

```text
what is reused
what is changed
why the change is necessary for AniGroom
```

This applies to:

```text
rendering
root initialization
root surface coordinates
orientation supervision
root neighborhood construction
densification and pruning
train/test evaluation
debug visualization
```

The previous hand-written renderer and old smoke-test pipeline are not formal
references. They may only provide lessons about useful mechanisms and failure
modes.

Forbidden in the formal Stage 1 implementation:

```text
hand-written renderer as the training backend
undocumented fallback behavior
ad hoc parameters without grooming meaning
latent-controlled xyz as the main geometry representation
carrier/base-fur decomposition story
darkness parameter
old face-residual global random root resampling
```

## 3. Required Plain `gsplat` Baseline

Before AniGroom Stage 1 training, run a plain `gsplat` baseline on the same
white tiger data.

Purpose:

```text
verify camera loading
verify image/mask loading
verify resolution and metric pipeline
verify train/test split
verify gsplat forward/backward
verify speed and memory
provide a sanity-check visual baseline
```

The baseline uses ordinary free Gaussian parameters:

```text
xyz
scale
rotation
opacity
color / SH
```

The baseline must not include AniGroom roots, strands, or custom renderer code.

## 4. Data and Train/Test Protocol

Training views are used for optimization. Test views are never used for
optimization.

Training supervision may include:

```text
RGB images
foreground masks
orientation maps
orientation confidence
camera matrices
optional head region annotations
optional SDF / mesh distance fields
```

Evaluation must disable all training-only randomization.

Minimum reported outputs:

```text
train RGB metrics
test RGB metrics, if test views exist
mask / alpha metrics
head-region metrics, if head regions exist
orientation metrics on supervised views
root count
Gaussian/sample count
training time
peak memory
turntable or held-out view renders
```

If the dataset has no official test views, a fixed held-out validation split
must be defined before training and kept unchanged.

## 5. Rendering Backend

The formal renderer is `gsplat`.

Pipeline:

```text
mesh-rooted groom parameters
-> strand samples
-> anisotropic Gaussian parameters
-> gsplat rasterization
-> RGB / alpha
-> losses
```

Gaussian parameters passed to `gsplat`:

```text
means
scales
rotations
opacities
colors
```

All Gaussian parameters must be generated by differentiable PyTorch operations.
`gsplat` provides gradients to Gaussian parameters, and autograd propagates
those gradients back to roots and groom parameters.

## 6. Mesh Roots

Each fur root is constrained to the input mesh surface.

Root representation:

```text
face_id
barycentric coordinate on that face
```

For a face with vertices `(v0, v1, v2)`:

```text
x_root = b0 * v0 + b1 * v1 + b2 * v2
b0 + b1 + b2 = 1
b0, b1, b2 >= 0
```

The first formal implementation uses face-local root motion:

```text
b = softmax(a)
x_root = b0 * v0 + b1 * v1 + b2 * v2
```

`a` is optimized. This gives differentiable movement inside the supporting
triangle while keeping roots on the mesh surface.

Cross-face differentiable movement is not included in Stage 1. Larger
redistribution is handled by densification and pruning.

## 7. Root Initialization

Initial roots are sampled from the mesh surface, not only from raw mesh
vertices.

Confirmed initialization:

```text
sample dense candidate points on mesh faces
select approximately uniform initial roots, e.g. FPS over candidates
store face_id and barycentric coordinate for each root
```

Initial root count starts at the `1e4` scale. Higher capacity is added by root
densification.

Root existence is represented by the root set itself:

```text
root exists -> one strand can be generated
root is pruned -> no strand exists
```

There is no separate root-existence or density variable in the first version.

## 8. Groom Parameters

Each root owns explicit groom parameters.

Confirmed Stage 1 parameters:

```text
length
root width
tip width
flow direction
bend / sag
stiffness
color
opacity
```

Parameter roles:

```text
length:
  strand length from the mesh surface.

root width / tip width:
  strand thickness profile and Gaussian scale.

flow direction:
  tangent-plane brushing direction.

bend / sag:
  static geometric bending, including gravity-like downward influence.

stiffness:
  resistance to bend and sag.

color:
  fur color.

opacity:
  Gaussian alpha along the strand.
```

Opacity controls rendered visibility. It is not a substitute for local root
count. Local root count is controlled by root initialization, densification,
and pruning.

## 9. Strand and Gaussian Generation

Each root generates one strand in the first version.

Generation path:

```text
root surface coordinate + groom parameters
-> strand curve samples
-> sample tangent directions
-> Gaussian means / scales / rotations / opacities / colors
-> gsplat
```

The strand generator should include static sag/gravity because training images
already contain gravity-shaped fur.

If the asset is later exported to an engine, the learned strand can be treated
as a rest groom. Engine-side simulation should then add only secondary motion
or use a less-sagged export preset to avoid double-counting gravity.

Adaptive strand/Gaussian sampling from the previous version is a useful
efficiency mechanism:

```text
short/simple strands -> fewer samples
long/curved strands -> more samples
```

The first implementation may start with fixed samples for clarity, but the
formal framework should preserve the path to adaptive sampling and record
sample-count statistics.

## 10. Orientation and Flow System

Flow is a first-class part of Stage 1.

Orientation maps are generated by:

```text
RGB image
-> grayscale
-> Difference-of-Gaussians filtering
-> Gabor filter bank
-> orientation + confidence
```

This is not Canny edge detection.

Useful mechanisms recovered from the previous version:

```text
orientation-map based flow initialization
flow hint prior
projected flow orientation loss
flow coherence regularization
confidence-masked orientation supervision
optional flow stabilization / lower learning rate after convergence
```

Formal behavior:

```text
use foreground/hair masks and orientation confidence
project strand tangents into image space
compare projected strand direction with target orientation
do not force unreliable pixels
regularize neighboring root flow on the root graph
```

## 11. Head Region Handling

The head is a special failure region and must not be hidden by full-body
averages.

Useful mechanisms recovered from the previous version:

```text
head-region metrics
head-region weighting for existing RGB / mask / orientation losses
head-prioritized root allocation or densification evidence
optional higher-resolution head parameter storage
```

This does not introduce unrelated loss families. It reweights existing losses
on the head region when head masks, boxes, or reliable projected regions are
available.

Logs must report head-region metrics separately.

## 12. Boundary and Detail Weighting

Boundary and high-detail regions are important and were useful in the previous
version.

Keep the following ideas:

```text
boundary-aware mask weighting
detail-weighted RGB loss
boundary evidence for root allocation
detail evidence for local capacity decisions
```

Formal behavior:

```text
derive weights from masks, image gradients, and orientation confidence
use weights on existing losses
use evidence only to select where more root capacity may be needed
do not introduce unrelated appearance decomposition
```

## 13. Random Mesh / Background Backing

Randomized mesh or background color is a training stabilization mechanism.

Purpose:

```text
prevent white fur on white background from being optimized as semi-transparent alpha
```

Rules:

```text
use only during training
disable during evaluation
do not present as a contribution
do not turn it into carrier/base-fur decomposition
```

## 14. Root Neighborhood Graph

The formal implementation requires a root neighborhood graph.

The same graph concept should support:

```text
groom parameter smoothness
flow coherence
root densification placement
pruning and local root-count control
local edit propagation
```

UV coordinates are only for storage and visualization. Core smoothness is
defined on mesh/root neighborhoods, not fragmented UV pixel adjacency.

The previous random same-face/adjacent-face edge sampling is not the formal
definition. The formal graph must be deterministic and reproducible.

## 15. Smoothness and Cleaning

Recovered useful mechanisms:

```text
root-graph groom smoothness
flow coherence
parameter range regularization
geometry prior for strand length / width / sag / stiffness
```

Purpose:

```text
reduce noisy, fragmented, overfit fur
keep parameters readable and editable
avoid UV-space speckle patterns becoming geometry
```

Smoothness must be applied on root/mesh neighborhoods. UV-space TV can be used
only as an auxiliary storage regularizer if needed; it is not the core
geometric smoothness mechanism.

## 16. Root Densification

Densification adds roots, not raw Gaussians.

The procedure has two separate parts:

```text
1. parent-root selection
2. local new-root placement
```

### 16.1 Parent-Root Selection

Parent-root selection decides which existing roots need more local capacity.

Signals may include:

```text
per-Gaussian visibility / gradient / contribution aggregated by root_id
image residual evidence aggregated by root_id
orientation failure evidence aggregated by root_id
boundary evidence aggregated by root_id
head-region evidence aggregated by root_id
```

These signals only select parent roots. They do not directly create new roots
and must not be used for global random resampling.

### 16.2 Local New-Root Placement

After parent roots are selected, new roots are placed locally around selected
parents.

Confirmed principle:

```text
selected parent root
-> inspect local root distribution around the parent
-> place new root(s) in nearby under-covered surface positions
```

The exact placement rule must remain simple, fast, and tied to the root
neighborhood graph. It must not become a global search algorithm.

New roots inherit or locally interpolate groom parameters from the parent and
its neighborhood. Root tensors, masks, optimizer state, and root-id mappings
must be synchronized after insertion.

Forbidden:

```text
old face-residual weighted global random root resampling
```

## 17. Root Pruning

Pruning removes roots with consistently low evidence.

Prune candidates:

```text
invisible across training views
low contribution for a sustained interval
low effective opacity
```

Pruning must not start before the training views have been observed at least
once.

Prune start:

```text
prune_start = max(one_train_pass_iterations, configured_prune_warmup)
```

Densification and pruning are synchronized in the same structure-update step:

```text
1. compute parent-root selection signals
2. add new roots
3. expand/rebuild root tensors, masks, optimizer state, and root-id mappings
4. compute prune mask on the post-densification root set
5. remove pruned roots
6. rebuild final tensors, masks, optimizer state, and root-id mappings
```

Densification happens before pruning. Low-contribution roots should not be
selected for densification, so the two operations should not conflict. The
important implementation detail is index consistency: any mask computed before
adding roots is invalid after roots are inserted.

## 18. Stage 1 Losses

Confirmed loss families:

```text
RGB reconstruction loss
mask / alpha loss
orientation / flow loss
root-graph smoothness
flow coherence
length / width / opacity regularization
bend / sag / stiffness regularization
boundary/detail weighted variants of existing losses
head-region weighted variants of existing losses
penetration or surface constraint, if available
```

Head, boundary, and detail terms are weights or evidence applied to existing
losses. They are not separate unrelated objectives.

## 20. Implementation Boundary

This document is complete enough to guide the next implementation stage, but
implementation should not start until the framework is reviewed and accepted.

The next step is not to write training code immediately. The next step is to
review this technical framework, confirm all module boundaries, and then create
a clean implementation plan.
